# Focus checks and correction memory boundaries

The Voice Keyboard Engine should not pretend it can reliably detect every missing focused input field through generic text injection alone. The current Input Environment can insert text, read an Explicit Selection, and maintain Tracked Segment safety, but reliable focus detection is platform-specific and often application-specific. A future "ask whether to paste" behavior must be implemented behind the Input Environment seam using platform accessibility APIs or desktop host cooperation, and should report best-effort confidence rather than a universal guarantee.

The Voice Keyboard Engine should also keep correction memory separate from Reusable Text Memory. Reusable Text Memory stores short user-provided snippets for later insertion into the Input Environment. Correction memory is a different rule set: it may need user glossary entries, hotwords, provider prompt injection, or correction pairs that influence a Speech Interpretation Provider. Those rules should be designed as a separate module before implementation, not folded into generic Reusable Text Operation behavior.
